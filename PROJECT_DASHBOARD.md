## Shared State Files

All threads should keep these files consistent:

```text
docs/status/PROJECT_STATUS.md
docs/status/DECISION_LOG.md
docs/status/EXPERIMENT_LOG.md
```

Rules:

- `PROJECT_STATUS.md` is the compact truth of where the project currently stands.
- `DECISION_LOG.md` records choices that should not be re-litigated without new evidence.
- `EXPERIMENT_LOG.md` records important results and what they imply.
- Detailed logs stay in `experiments/`; summaries are generated by `LLM_code/summarize_experiments.py`.

## Cross-Thread Sync Protocol

At the end of any substantial thread, update at least one shared state file:

| Change Type | Required Update |
| --- | --- |
| New experiment result | Add interpreted result to `docs/status/EXPERIMENT_LOG.md` |
| New method decision | Add decision and reason to `docs/status/DECISION_LOG.md` |
| New project direction or risk | Update `docs/status/PROJECT_STATUS.md` |
| New code behavior or runner flag | Update `docs/threads/02_code.md` and the relevant run doc |
| New data alignment finding | Update `docs/threads/04_data_alignment.md` |

Before starting a new thread:

```text
1. Read PROJECT_DASHBOARD.md.
2. Read the relevant docs/threads/*.md file.
3. Read PROJECT_STATUS.md.
4. If using experiment results, read EXPERIMENT_LOG.md or rerun summarize_experiments.py.
```

Before ending a thread:

```text
1. Record what changed.
2. Record what evidence supports it.
3. Record the next recommended action.
4. Push documentation changes when they affect future work.
```

## Current Technical Notes

- `USE_MM_PREFIX=False` does not read C manifest features.
- `USE_MM_PREFIX=True` now prefers `utterance_id` for C-manifest alignment.
- `GRADIENT_CHECKPOINTING=True` is available and reduces Qwen2.5 training memory substantially.
- IEMOCAP legacy + C-feature alignment was improved by adding stable `utterance_id`.
- Direct AV prefix should be treated as an ablation baseline, not the final method.
